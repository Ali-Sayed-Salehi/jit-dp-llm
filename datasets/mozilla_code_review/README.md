# Mozilla Code Review Dataset

This directory contains Mozilla Mercurial commits, model risk predictions, and
Phabricator code-review data linked at the per-commit level.

## Files

- `all_commits.jsonl`
  - One Mercurial commit per line.
  - Fields: `node`, `desc`, `date`, `parents`.
  - The file is expected to be parent-before-child ordered. The Conduit scripts
    use this order as the Mercurial graph order.

- `risk_predictions_eval.json`
  - Model predictions for the eval split.
  - Prediction rows are under `samples`.
  - Each row includes `commit_id`, `true_label`, `prediction`, and
    `confidence`.

- `risk_predictions_final_test.json`
  - Model predictions for the final-test split.
  - Prediction rows are under `results`.
  - Each row includes `commit_id`, `true_label`, `prediction`, and
    `confidence`.

- `per_commit_drevs.jsonl`
  - Generated by
    `data_extraction/conduit/get_per_commit_drevs.py`.
  - One row per commit with a trailing Mozilla Phabricator Differential
    Revision URL whose DREV is published and closed.
  - Top-level fields:
    - `commit_id`: Mercurial commit hash.
    - `dataset_split`: either `eval` or `final test`.
    - `drev`: raw Conduit DREV object returned by
      `differential.revision.search`.

- `per_commit_drev_transactions.jsonl`
  - Generated by
    `data_extraction/conduit/get_drevs_transactions.py`.
  - One row per `per_commit_drevs.jsonl` row.
  - Top-level fields:
    - `commit_id`: Mercurial commit hash.
    - `drev_id`: numeric Differential Revision id.
    - `dataset_split`: copied from `per_commit_drevs.jsonl`.
    - `transactions`: raw Conduit transaction objects returned by
      `transaction.search`.

## Split Assignment

`get_per_commit_drevs.py` loads both prediction files and sorts their
`commit_id` values by the parent-child order in `all_commits.jsonl`.

It then defines:

- eval boundary: first to last eval prediction commit in Mercurial order.
- final-test boundary: first to last final-test prediction commit in Mercurial
  order.

The script only scans commits from the eval start boundary through the
final-test end boundary. For each scanned commit:

- commits in the eval boundary get `dataset_split = "eval"`;
- commits in the final-test boundary get `dataset_split = "final test"`;
- if the boundary ranges overlap, direct membership in the prediction files is
  used first, then nearest prediction sample in Mercurial order is used as a
  deterministic tie-breaker.

This means a commit can receive a split label even if it is not itself present
in a prediction JSON file, as long as it falls inside one of the split
boundaries.

## Regenerating

From the repo root:

```bash
source venv/bin/activate

python data_extraction/conduit/get_per_commit_drevs.py
python data_extraction/conduit/get_drevs_transactions.py
```

For a bounded smoke test:

```bash
python data_extraction/conduit/get_per_commit_drevs.py --debug
python data_extraction/conduit/get_drevs_transactions.py --debug
```

The full run uses Mozilla Phabricator Conduit and requires
`CONDUIT_API_TOKEN` in `secrets/.env`.
